Modularization Proposal — go-workflow-auditlog

Date: 2026-07-22 · Author: Crush

Executive Summary

Split the single-package library into a core module (auditlog) that owns the workflow event capture, report model, JSON/NDJSON serialization, and replay; and a visualization sub-module (viz) that owns diagram, table, tree, and HTML dashboard rendering. This removes the heavy go-output dependency from the core module, gives consumers a smaller import surface, and makes the dependency graph acyclic.

Current State

  • Single module: github.com/larsartmann/go-workflow-auditlog
  • Go 1.26.4, no go.work
  • Package auditlog contains ~38 production Go files plus tests
  • Core domain and visualization code share one package and one go.mod
  • go-output and its sub-modules are required by the core module because rendering methods live on Auditor and WorkflowReport

DAG Verification

The proposed dependency graph is acyclic:

viz ──depends on──> core
example ──depends on──> core, viz

Core does not import viz. No circular dependencies.

Replace / Workspace Strategy

Use the dual strategy recommended for multi-module Go repositories:

  • go.work at the repo root lists . and ./viz for development.
  • viz/go.mod contains replace github.com/larsartmann/go-workflow-auditlog => .. so it builds standalone with GOWORK=off.
  • Internal module references are pinned to v0.0.0 to avoid pseudo-version churn.

Test Dependency Isolation

Shared workflow fixtures currently live in auditlog_test.go (external test package). After the split, both core and viz tests need them. They will be extracted into a testhelpers package inside the core module. The core module still does not depend on viz.

API Changes

Visualization methods currently on Auditor and WorkflowReport become package-level functions in viz that accept a WorkflowReport:

// Before
report.WriteMermaid(w)
audit.ExportHTML("report.html")

// After
viz.WriteMermaid(report, w)
viz.ExportHTML(report, "report.html")

Core JSON/NDJSON exports remain on WorkflowReport and Auditor because they are serialization, not visualization.

Error Type Placement

  • ErrRenderFailed stays in core: it is used by both JSON marshaling and visualization, and core is the lower layer.
  • ErrExportWriteFailed stays in core: used by all file exports including core JSON/NDJSON.
  • Visualization code wraps renderer errors with ErrRenderFailed from core.

Versioning Strategy

Shared version tags. The project is pre-1.0 and alpha; both modules will ship together under a single v0.x.x git tag. The sub-module path is .../viz, not a /v2 major-version suffix.

Breaking Change Analysis

This is a breaking change for consumers using diagram/table/tree/HTML exports:

  • Remove Auditor.WriteMermaid, WritePlantUML, WriteGraphviz, WriteD2, WriteTable, WriteTree, WriteHTML, and their Export* and Write*String variants.
  • Remove WorkflowReport methods of the same names.
  • Move WithDirection, WithColumns, column constants, and TableColumn to the viz package.

Consumers that only capture events and export JSON/NDJSON are unaffected.

Migration Strategy

  1. Create testhelpers package and move shared fixtures; update core tests.
  2. Create viz directory, viz/go.mod, and public viz API.
  3. Move rendering files and their tests into viz; convert methods to functions.
  4. Remove visualization methods from core Auditor and WorkflowReport; remove go-output from core go.mod.
  5. Move example to viz/example and update imports.
  6. Create go.work and update go.work.sum.
  7. Update CI, goreleaser, README, and AGENTS.md.
  8. Run full test suite and GOWORK=off per-module verification.
