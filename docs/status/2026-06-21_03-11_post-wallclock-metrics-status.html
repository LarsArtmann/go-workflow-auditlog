05E · What We Should Improve

!
Single construction path is a strength — protect it buildReportFromCore() → finalizeDenormalized() is the only place aggregates derive. New report builders (Filtered, ReplayEvents) already route through it. Any future path MUST too.
!
Move Duration() to report.go The wall-clock method sits in diff.go alongside diff-only helpers. report.go owns WorkflowReport and is where consumers look. Cohesion fix, low risk.
!
Make step names deterministic by default flow.String(step) returns pointer addresses. The library documents this as a known limitation but could offer a Name(step) helper that falls back to type name when String() is unset.
!
Coverage is 93.2% — push to 95%+ The new validateStatusCounts branches and omitempty fields are the likely uncovered lines. Add targeted tests rather than chasing coverage for its own sake.
!
Status reports as HTML, not Markdown The last two status reports were .md files; the skill mandates self-contained HTML dashboards for snapshots. Snapshots get the visual treatment; living docs stay Markdown.
!
Consider a typed RunID RunID is a plain string today. A branded string type (RunID) would make impossible states unrepresentable and improve API ergonomics without breaking JSON.

07G · The One Question I Can’t Answer

Blocking question

Is the BuildFlow audit-log JSON consumed by an automated system, or only read by humans?

I added 6 new fields to the public WorkflowReport JSON shape and changed the semantics of Summary() and Diff().DurationDelta from sum-of-durations to wall-clock. For a strict JSON consumer (a dashboard, an alerting rule, a regression gate) the added fields are safe, but the semantic change to existing methods could quietly flip a regression signal from “no change” to “improved”.

If BuildFlow (or any other consumer) parses this report programmatically, we should: (a) confirm the consumer keys off TotalDurationMs (unchanged, safe) and not off Summary() output (changed), and (b) decide whether the semantic change warrants a v0.2.0 minor bump with a migration note.

I cannot determine this from the repo — it requires knowing who reads workflow-audit-log.json downstream. This directly gates the release-strategy decision (patch vs minor) and the CHANGELOG framing.
