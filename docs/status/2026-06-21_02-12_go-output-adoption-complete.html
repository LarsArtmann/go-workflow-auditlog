DTotally Fucked Up!

Critical

Dependency bloat: from 2 to 12 direct deps, ~35 transitive

go.mod

The library went from 2 direct dependencies (go-workflow + backoff) to 12 direct + ~35 transitive. The table module pulls in charm.land/lipgloss/v2 + 8 charmbracelet transitives. The serialization module pulls in go-faster/yaml + go-faster/jx + go-faster/errors + segmentio/asm. This contradicts the original "zero heavy deps" philosophy. An auditlog library should not transitively require a TUI styling framework.

Root cause: table.go blank-imports go-output/table (which requires lipgloss) and go-output/serialization (which requires yaml). These are auto-registered via init() so RenderTableData can dispatch to them. Every consumer of the auditlog library now pulls in lipgloss + yaml even if they only want JSON diagrams.
Important

go-output version skew: root v0.17.0 vs sub-modules v0.13.0

go.mod

go-output root is at v0.17.0 but all sub-modules are tagged at v0.13.0. This works today because MVS resolves the sub-modules independently, but it's a fragile arrangement. If go-output changes its module structure or drops the v0.13.0 tags, this breaks.

EWhat We Should Improve

1. Make heavy deps optional

The lipgloss and yaml dependencies should NOT be forced on every consumer. Options: (a) split auditlog into core + visualization sub-modules; (b) use build tags for optional format support; (c) lazy-import via interface satisfaction rather than init() blank imports. The current blank-import pattern in table.go is the root cause.

2. Deduplicate buildGraph patterns

diagram.go:buildGraph(), d2.go:buildD2Graph(), and tree.go:buildTreeNodes() share the same "iterate steps, deduplicate by name, add nodes + edges" pattern. Run art-dupl -t 15 to verify and extract a shared traversal helper if warranted.

3. Error path test coverage

Most Write* methods sit at 83% coverage — the error paths (writer failure, render failure) are untested. Add tests that inject failing io.Writers and verify error wrapping.

4. Table export coupling

table.go imports 4 sub-modules via blank imports. This means ANY consumer of auditlog gets ALL 16 table format renderers compiled in, even if they only use JSON. Consider a tableformats/ sub-package that consumers can import opt-in.

5. Report analytics are well-tested

computePeakConcurrency (88.9%), computeCriticalPathDuration (96.2%), buildFailureReason (94.4%) — solid coverage. The analytics logic is clean and single-purpose. Good work.

GThe #1 Question I Can't Answer Myself

Should this library remain single-module with all go-output sub-modules as direct dependencies, or should it split into a core auditlog module (JSON/NDJSON only) and a visualization module (diagrams/tables/trees)?

The original design philosophy was "zero heavy deps" — 2 direct dependencies. The go-output adoption adds 12 direct + ~35 transitive dependencies including lipgloss (TUI styling), yaml, and toml parsers. This is fine if auditlog is meant to be a full-featured observability tool. But if consumers want a lightweight audit trail library (just JSON/NDJSON event capture), they now pay a heavy dependency tax for features they may not use.

The tradeoff: splitting into sub-modules (like go-output itself does) keeps the core dependency-light but adds module management complexity. Keeping single-module is simpler but forces all consumers to carry lipgloss + yaml. This is an architectural direction decision, not a technical one — I can implement either path, but I can't decide which is right for your intended audience.
