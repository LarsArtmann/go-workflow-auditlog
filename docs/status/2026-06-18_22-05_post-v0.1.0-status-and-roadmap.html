One-liner: The library is in good shape. v0.1.0 was released two days ago with full documentation refresh, security disclosure process, and changelog. Today we shipped one bug fix (StepInfo.Error now reflects the final outcome only, not the last attempt's error). The next major push is API ergonomics, observability hooks, and one structural refactor (split the package).

What's strong

+
Zero external runtime deps Single Go module, depends only on Azure/go-workflow and cenkalti/backoff/v4. No transitive risk surface to audit.
+
Single construction path buildReportFromCore is the only way a WorkflowReport is built — no risk of denormalized fields drifting from core data.
+
Honest schema + validation WorkflowSucceeded requires zero pending/running steps. Validate() catches status/error drift. Comments don't lie.
+
Comprehensive export surface JSON, NDJSON, Mermaid, PlantUML, Graphviz DOT, replay, diff, index, filter, load. 6 ways to consume the same data, all round-trippable.

What's hurting

−
Single package, 17 source files Long compile times when editing. Boundary between core, serialization, and exporters is invisible. Hard to depend on a subset.
−
Coverage has a ceiling around 95% Defensive nil checks and a couple of error-formatting branches in recordAfterStep and attach.go remain untested.
−
No public benchmarks in CI Benchmarks exist (benchmarks_test.go) but aren't run on PRs. We have no performance regression safety net.

aFully Done

Core event capture

Before/After callbacks captured per attempt via State.MergeConfig injection. Single mutex guards all mutable state; onEvent callback always fires outside the lock.

DAG-aware Snapshot

Snapshot reads StateOf/UpstreamOf post-run, including sub-workflow inner steps via flow.Traverse. Captures skipped/canceled steps that bypass callbacks.

Export: JSON, NDJSON, Mermaid, PlantUML, Graphviz DOT

5 distinct export formats, all using the shared writeDiagram engine for graph output. NDJSON reader handles malformed lines with line-numbered errors.

Query surface: Filter, Index, Diff

Filtered with ReportOptions (steps-by-status, time range, drop). ReportIndex for O(1) name/ID lookup. Diff for run-over-run comparison with wall-clock Duration + Summary.

Replay & Load

ReplayEvents reconstructs a Report from an event stream, preserving RunID and assigning StepIDs. LoadReport / LoadReportFromReader / LoadReportFromBytes for round-trip.

Validation & status drift detection

WorkflowReport.Validate() catches event/step count mismatches and status drift (stored Status != derived Status). Sentinel errors for programmatic handling.

Example pipeline + goreleaser

Demo binary shows fetch → validate → transform → save with retry. .goreleaser.yml automates GitHub releases with grouped changelog + demo artifact.

v0.1.0 documentation + security policy

README, CHANGELOG, FEATURES, TODO_LIST, ROADMAP, AGENTS.md all current. SECURITY.md with private vulnerability reporting. Status reports for the release prep.

StepInfo.Error = final outcome (fixed today)

Regression test: succeeded-after-retry step now has nil Error. Per-attempt errors remain in the Event stream. Invariant documented in AGENTS.md gotchas.

cNot Started

Structured logging (slog) integration

The Auditor accepts a generic onEvent func(Event) callback but doesn't integrate with log/slog. A SlogHandler adapter that emits one structured log record per event would unlock native observability pipelines.

OpenTelemetry trace correlation

RunID is a 128-bit random hex. No built-in way to map it to a trace ID. The recorder already has the right shape — would need a small adapter that injects trace.SpanFromContext into the RunID at Attach time, and emits OTel events.

Prometheus / OpenMetrics exporter

Per-step histograms for duration, counters for status transitions, gauges for in-flight attempts. ~150 lines plus a Collect method. No design exists yet.

HTML report exporter

Mermaid is great for DAGs but a self-contained HTML report (timeline + DAG + step detail table) would be a strong differentiator. We have the template skill already.

Sub-workflow composition tests

subworkflow_test.go exists but covers only one pattern. No test for: deeply nested sub-workflows, sub-workflow with its own retries, sub-workflow error propagation.

Property-based tests

All tests are example-based. testing/quick or pgregory.net/rapid could generate random workflow topologies + step result sequences and assert report invariants (event count == 2×attempt count, error/status coherence, etc.).

Performance regression CI

Benchmarks exist in benchmarks_test.go but aren't gated in CI. Need a GitHub Action that runs go test -bench=. -benchmem and posts a comparison comment on PRs.

Split single package into core / exporters

17 files in auditlog is too many. Natural split: auditlog (core types, recorder, report) + auditlog/export (JSON, NDJSON) + auditlog/diagram (Mermaid, PlantUML, DOT). Would need to update all imports; mechanical but verbose.

v0.2.0 API stabilization

Once we have external users, we need a v0.2.0 with breaking-change budget. Currently the API can still evolve freely (good for ALPHA), but several types will need to firm up: Config fields, ReportOption semantics, error types.

WorkflowReport → proto / flatbuffers

JSON is fine for files, expensive for IPC. If we ever want to ship reports over gRPC or share them between processes, a binary schema would help. Not urgent.

dTotally Fucked Up

Critical

go-workflow v0.1.13 data race in DefaultRetryOption.Backoff

upstream: github.com/Azure/go-workflow

Workaround in place: every test creates o.Backoff = backoff.NewExponentialBackOff() (fresh instance). Without this workaround, -race explodes on any concurrent retry. We can't fix it in our library; we're at the mercy of upstream. Mitigation: pin go-workflow and document the trap. Risk: every contributor must remember to use retryOpts() in tests, not the bare RetryOption.

Watch out New contributors will hit this within their first retry test. AGENTS.md documents it but it's still a foot-gun. Consider adding a CI test that runs the example under -race and fails if the backoff is shared.
Critical

Step name collisions are silently ambiguous

recorder.go:stepKey / report_builder.go:buildDependentsMapLocked

Internal step identity uses the flow.Steper pointer (correct, unique). But the JSON step_name field uses flow.String(step), which is non-deterministic for steps that don't implement String() (yields *TypeName(0xpointer)). Two steps with identical types and no String() produce non-matching step_name strings across runs, but matching ones with the same pointer can produce identical-but-unrelated names within one report. buildDependentsMapLocked uses name → pointer mapping for reverse deps, so a name collision breaks the dependent graph.

Fix Use pointer identity (or StepID) as the canonical key everywhere internally; emit the human name only in the JSON output. We've documented this as a known limitation but haven't refactored.
Critical

flow.StatusFromError semantics are undocumented & fragile

recorder.go:fromErrorToStatus

We use flow.StatusFromError(err) which returns capitalized strings ("Succeeded", "Failed", "Canceled", "Skipped") that we then lowercase. If upstream changes the mapping or the casing, our status enum breaks silently. The conversion lives in types.go:fromFlowStatus as a switch with a default → Pending that hides breakage.

Watch out No test asserts that flow.StatusFromError returns the specific strings we expect. A name change in upstream would break without any test failure.
Important

No upper bound on RunID / no RunID validation

plugin.go / runid.go

newRunID() returns 32-char hex from crypto/rand. If a user passes their own RunID via Config.RunID, we never validate it. Empty string is silently accepted. Empty RunID propagates to every Event's run_id field, breaking correlation.

Fix In NewRecorder: if RunID is provided, validate non-empty and warn (or error) on values that look like they will collide (e.g. all zeros). Empty → auto-generate.
Important

Sequence counter is atomic but events can be observed out of order

recorder.go:onEvent / AGENTS.md line 65-69

AGENTS.md says "delivery order is not guaranteed to match event Sequence (sort if needed)" — but the only way to sort is Sequence itself, and Sequence is assigned in lock-step with the event creation, not with the goroutine ordering. The OnEvent callback fires from parallel step goroutines; if the recorder is busy when goroutine A finishes (e.g. holding the lock for goroutine B), goroutine A's onEvent call is delayed. The Sequence is still monotonic in the recorder, but a consumer that sorts by Sequence and then processes in goroutine-launch order will see inversion. AGENTS.md is honest about this but the field name "Sequence" is misleading.

Watch out Don't rename "Sequence" — it's load-bearing in the JSON schema. But consider adding RecordedAt time.Time alongside (already present) and clearly document that wall-clock timestamp, not Sequence, is the source of truth for ordering.
Important

BenchmarkRetryWithAudit is 1.5 SECONDS (likely test bleed, not real perf)

benchmarks_test.go

All other benchmarks are sub-millisecond. BenchmarkRetryWithAudit-32 = 1,507,983,375 ns/op (1.5s per iteration). The retry test uses real backoff with sleeps, so the benchmark is measuring wall-clock sleep, not the audit overhead. It is useless as a perf signal.

Fix Either use a no-op backoff (constant 0 duration) for the benchmark, or split it into BenchmarkRetryOverhead (no real backoff) and BenchmarkRetryEndToEnd (real backoff, not for regression comparison).

gMy #1 Open Question

Should we add a LastError (or TransientErrors) field to StepInfo for the forensic use case, or stay minimal?

Today's fix (Error = final outcome only) is the right call for the default semantic. But it does lose information: a user looking at a succeeded step has no fast way to see "this step transiently failed twice before succeeding" without scanning the event stream and filtering for IsAttemptEnd() && HasError().

I see three options and I don't know which is best for this project's audience:

  1. Stay minimal. Error = final outcome. Users who want per-attempt history go to the Event stream. Clean, single source of truth. But: extra work for the common "did this step ever fail?" question.
  2. Add TransientErrors []string. All non-final attempt error messages, in order. Final outcome is still Error. Preserves forensics, doesn't conflate with final error. Cost: ~5 lines, one extra test, one extra JSON field. Slight schema bump (additive only — v0.2.0 is fine).
  3. Add LastAttempt int + reuse Error for final, AttemptErrors map[int]string for the rest. Most expressive, most schema surface, most complexity. Probably overkill for v0.x.

What's the right call depends on: who the target users are, whether this library is meant for human-readability (the example) or machine-aggregation (a metrics pipeline), and whether we expect StepInfo to be queried with SQL-like filters. I can't decide this alone — it requires product judgment about audience.

My gut Option 2. TransientErrors []string is additive, cheap, and answers the "did this step ever fail?" question in O(1) JSON lookups. But I'd want to hear from a real user first.
