## D) TOTALLY FUCKED UP 💥

### D1. Golden file corruption (FIXED)

**What happened**: The first commit (`7e928b6`) included a golden file that was pretty-printed by an external formatter (lowercase `<!doctype html>`, 2-space indentation, XHTML self-closing tags). The `renderHTML` function produces uppercase `<!DOCTYPE html>` with no pretty-printing. The golden test silently failed after the commit.

**Root cause**: A pre-commit formatter (likely prettier via buildflow in a previous run) reformatted the `.html` file before it was committed. The test passed when I ran it before committing because the file hadn't been formatted yet.

**Fix applied**: Re-generated the golden file (`UPDATE_GOLDEN=1`), committed with the correct 56609-byte version. Verified the pre-commit hook does NOT reformat it on subsequent commits.

**Lesson**: Always re-run the golden test AFTER committing to verify the committed version matches.

### D2. 1300 lines of CSS/JS as Go string constants

**What I did**: Embedded 550 lines of CSS and 714 lines of JavaScript as Go backtick string constants in `html_render.go`. That's 88% of the file content as non-Go code.

**Why it's terrible**:
- No syntax highlighting in any editor
- No CSS/JS linting
- No formatting (gofumpt can't help)
- `//nolint:lll` suppression at file level
- Impossible to debug CSS/JS issues without modifying Go code
- The `renderHTML` function is 128 lines with `//nolint:funlen,varnamelen`

**Should have done**: Use `go:embed` to embed `dashboard.css` and `dashboard.js` as separate files. This is stdlib, zero new dependencies, and gives proper syntax highlighting + linting.

## E) WHAT WE SHOULD IMPROVE

### E1. Architecture: Extract CSS/JS to embedded files (`go:embed`)

**Current**: 1264 lines of CSS+JS as Go string constants in `html_render.go`
**Target**: `html_render.go` shrinks to ~100 lines of Go; CSS/JS live as `.css`/`.js` files with proper tooling support

### E2. Architecture: Deduplicate tree/graph logic

**Split brain #1**: Tree root detection logic exists in BOTH `tree.go` (Go) and `html_render.go` (JS). Same algorithm (`Dependencies.length == 0`), two languages.

**Split brain #2**: Status color mapping exists in THREE places:
- `types.go`: `stepStatusMeta` (FillColor/FontColor hex values)
- `html_render.go` CSS: `--success`, `--error`, etc. CSS variables
- `html_render.go` JS: `statusColors` map in `renderGraph()`

**Fix**: The JS can read colors from CSS variables (`getComputedStyle`). The tree logic duplication is acceptable (Go renders server-side, JS renders client-side) but should be documented.

### E3. Testing: Add HTML to API symmetry tests

`output_test.go` has table-driven tests for `Export*` and `Write*String` methods. HTML dashboard is missing from both. Add entries to `TestWorkflowReport_ExportMethods` and `TestAuditor_WriteStringMethods`.

### E4. Testing: Test replayed report HTML export

`ReplayEvents` produces a valid `WorkflowReport`. The HTML export should work from replayed reports. Currently untested.

### E5. Type model: Consider structured step type metadata

Currently `StepType` is a raw string (the Go struct type name). The HTML dashboard shows it but can't provide icons/labels/colors per type. Could add an optional `StepTypeRegistry` that users populate for richer display.

### E6. Consider `html/template` instead of raw string building

The `renderHTML` function is essentially a manual template engine (`sb.WriteString` × 20+). Go's stdlib `html/template` provides auto-escaping, which would eliminate the need for the manual `esc()` JS function and the JSON script tag injection pattern. However, the current approach is proven and XSS-tested, so this is a "nice to have" not a "must fix".
