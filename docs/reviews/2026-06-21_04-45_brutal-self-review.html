02What's Stupid

Split Brain

stepRecord (recorder.go) vs replayStep (replay.go)

recorder.go:30-52 · replay.go:13-22

Two structs track the same domain concept (accumulated step state during execution/replay) with nearly identical fields:

stepRecord (recorder.go)
replayStep (replay.go)
type stepRecord struct {
  StepRef
  stepID       int
  attemptCount int
  startedAt    *time.Time
  finishedAt   *time.Time
  durationMs   *float64
  attemptErr   *string
  status       StepStatus
  // + live-only fields
}
type replayStep struct {
  StepRef
  attemptCount int
  startedAt    *time.Time
  finishedAt   *time.Time
  durationMs   *float64
  attemptErr   *string
  status       StepStatus
}

replayStep is a strict subset of stepRecord minus the live-only fields (pendingAttempts, maxAttempts, hasRetry, hasTimeout, dependencies). The conversion logic from each to StepInfo is duplicated. If we add a field to one and forget the other, replay diverges from live capture silently.

Fix: Extract a shared stepAccumulator struct containing the common fields. Both stepRecord (live) and replayStep (replay) embed it. One conversion path to StepInfo.

Stringly Typed

RunID is a plain string everywhere

event.go:9 · report.go:34 · plugin.go:30

RunID is string in Event, WorkflowReport, Config, and 4+ method signatures. Nothing stops you from passing a WorkflowID where a RunID is expected. The how-to-golang policy mandates branded types for domain identifiers. This is the #1 type-model defect.

Fix: type RunID string with MarshalText/ UnmarshalText for JSON compatibility. Compile-time safety, zero JSON break. The report flagged this; I deferred it.

03What Could Be Better

encoding/json v1 on Go 1.26

The project uses encoding/json (v1) in 5 files. The how-to-golang policy says "Go 1.25+ → encoding/json/v2." I checked — v2 is not yet in the Go 1.26.3 stdlib (the proposal exists but isn't shipped). So this is a policy aspiration, not an actionable violation today. Document it and revisit when Go ships v2.

Error as *string instead of error

Event.Error and StepInfo.Error are *string, not error. This was a deliberate choice for JSON serialization (errors don't serialize), but it means consumers lose errors.Is/errors.As capability. The error is a dead string, not a structured value.

Tradeoff: Changing this would break the JSON schema. For an ALPHA library this is acceptable, but it's a bigger change than fits this session. Ticket it.

*float64 for DurationMs — pointer boilerplate

DurationMs *float64 appears in 5 places (Event, StepInfo, stepRecord, replayStep, local in recordAfterStep). The pointer distinguishes "not measured" (nil) from "zero" (0.0). But it forces heap allocation and nil-checking everywhere. A typed wrapper (type Millis float64 with a sentinel, or time.Duration with zero-as-absent) would be cleaner.

Deprecated aliases add maintenance surface

I added 6 deprecated methods (WriteReportJSON, WriteEventsNDJSON, ExportToFile, ExportEventsToNDJSON). For an ALPHA library that hasn't tagged v0.2.0 yet, this is premature backward compat. The old names could just be renamed. But since external tests use them (the test files call ExportToFile etc.), removing would break consumers. Verdict: keep, but document a removal timeline.

04Type Model Assessment

What's good

StepStatus is a proper typed enum with Label()/Icon()/Color() methods and a centralized metadata map. EventType and Phase follow the same pattern. StepRef is embedded for JSON flattening. The single-construction-path (buildReportFromCore) is excellent.

What needs fixing (this session)

  • RunID: plain string → branded type
  • stepRecord/replayStep: split brain → shared accumulator
  • computeWallClockDurationMs: stranded → move to report.go

What to consider later

  • StepID: plain int, not branded. Low risk since it's internal.
  • SchemaVersion: plain string "0.1.0". Should match semver module tag.
  • Error *string: loses structured error info. Schema break to fix.
  • DurationMs *float64: heap-allocated nullable. Consider time.Duration.
