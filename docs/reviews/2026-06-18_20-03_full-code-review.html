01Critical Bugs (Fixed)

Fixed

Validate() status-drift check was dead code

report.go:50-56

The condition step.Status != derived && step.Status.IsTerminal() could never fail: when status is terminal, DeriveStatus() returns the same value, so the left side of the && is always false. The stated intent — detect drift between stored status and error pointer — was unreachable. No test exercised the drift path, so the bug went unnoticed.

Fix Removed the && step.Status.IsTerminal() guard. Now fires whenever stored status disagrees with derived. Added TestReport_Validate_StatusDrift.
Fixed

Diff() output was non-deterministic

diff.go:48-63

Iterating theirs / ours maps produced randomized AddedSteps / RemovedSteps / StatusChanged ordering. Two diffs over identical inputs could yield different slice orders — a regression test's worst enemy.

Fix Collect names into slices and slices.Sort them before emitting StepDiff entries. StepDiff also gained an OldStatus field so callers can see from→to transitions. Added TestDiff_DeterministicOrdering.
Fixed

WorkflowSucceeded lied during mid-execution

report_builder.go:143-144

WorkflowSucceeded = FailedCount == 0 && CanceledCount == 0 returned true if no step had failed, even when steps were still pending or running. A report built mid-flight would have reported success.

Fix Added a pendingOrRunning counter to the aggregate builder; the flag is now false unless every step has reached a successful terminal state. Added TestWorkflowSucceeded_PendingStepsIsFalse.
Fixed

NDJSON error line number counted events, not lines

ndjson.go:45

fmt.Errorf("ndjson line %d", len(events)+1) reported the event count rather than the actual file line. A malformed line following blank lines would have pointed at the wrong place.

Fix Track a real lineNum counter (incremented per Scan, including blank lines). Added TestReadEvents_LineNumberInError.
Fixed

recorder.durationMs clobbered by stray AfterStep

recorder.go:156

rec.durationMs = durationMs unconditionally overwrote the prior attempt's measured duration. If a step's AfterStep fired without a matching BeforeStep (which go-workflow can do for edge cases), durationMs would become nil, silently erasing a previously valid duration.

Fix Only overwrite when durationMs != nil. The previous value is preserved.
Fixed

CI was broken: Go 1.23 vs go.mod 1.26.3, lint v1 vs config v2

.github/workflows/ci.yml + go.mod + .golangci.yml

go.mod requires Go 1.26.3 but CI installed 1.23 — the test job would have failed at go build. The lint job installed golangci-lint v1.62 while the config is v2 format. Worse: depguard's allow list used the unsupported $module variable, so every import of github.com/Azure/go-workflow and even the project's own package was flagged.

Fix Bumped CI to go-version: "1.26" and golangci-lint v2.4.0. Rewrote .golangci.yml: explicit depguard allow list, snake_case tagliatelle, scoped exhaustruct, narrowed varnamelen, etc.

05Strengths

+
Single-lock concurrency model One sync.RWMutex guards all mutable state; onEvent fires outside the lock. Simple, correct, easy to reason about.
+
Single Report construction path buildReportFromCore() is the only place denormalized aggregates are derived. BuildReport, Filtered, and ReplayEvents all route through it — no drift between paths.
+
Strong type-safe enums EventType, Phase, StepStatus are typed strings with display metadata tables. Impossible-to-misuse API.
+
Round-trip NDJSON / JSON / Replay contracts Events serialize to NDJSON, deserialize via ReadEvents, and reconstruct via ReplayEvents. Reports serialize to JSON, load via LoadReport*. Tested both ways.
+
External test package Tests live in package auditlog_test — guarantees tests exercise the public API only, catching accidental reliance on internals.
+
Functional-option filtering ReportOption pattern (WithStepsByName, WithStepsByStatus, WithEventsByType, WithTimeRange) is composable and idiomatic Go.
+
Diagram engine is properly factored diagramFormatter interface + shared writeDiagram means Mermaid and PlantUML are thin adapters. Adding DOT/D2 would be a 30-line change.

Remaining Recommendations (Not Fixed)

WorkflowReport has no RunID

Only WorkflowID is present. Two runs of the same workflow cannot be distinguished in storage. Consider adding a RunID (UUID) populated at New() time and serialized in the report.

Recorder.sequence uses *atomic.Int64 unnecessarily

Recorder is always passed by pointer. The indirection through *atomic.Int64 + newSequenceCounter() helper is historical; a value field would work and remove one allocation.

Sentinel errors are unexported

errReportEventCountMismatch, errReportStepCountMismatch, errReportStatusDrift, errWorkflowIDPathSep are package-private. Callers cannot use errors.Is to branch on them. Consider exporting them as Err… for a 1.0 release.

Event.Phase is redundant with EventType

Every AttemptStart event has Phase=Before; every AttemptEnd has Phase=After. Phase exists in the JSON schema for clarity but carries no information. Worth documenting or dropping in a future schema bump.

EventsByStep / EventsByType are O(N) per query

Acceptable for typical sizes, but a BuildReport-time index (map from name/type → []int) would make repeated queries O(1).

Step name collisions still possible

Two steps with the same String() produce the same step_name in JSON. Internal tracking uses the flow.Steper pointer so it's correct, but the JSON output is ambiguous. Consider a doc note + optional StepID field.
